# ArticleWriterAgentとWorkSummaryAgentのエビデンス収集を6ステップに体系化した

## 対象読者

- ArticleWriterAgent や WorkSummaryAgent を使って記事・日報を自動生成しているエンジニア
- 「生成された記事にいくつかの変更が抜けていた」という経験がある人
- エージェントのエビデンス収集プロセスを改善したい人

## この記事が扱う範囲

- **扱う**: 旧来のエビデンス収集手順の問題点、6ステップへの刷新内容、両エージェントへの適用
- **扱わない**: Markdown 記事の執筆スタイル、日報のフォーマット

---

## 背景：単一 diff では抜け漏れが起きる

変更前の ArticleWriterAgent は、エビデンス収集を次の4ステップで行っていた。

```markdown
1. git --no-pager status --short でワークツリーの状態を確認
2. git --no-pager diff --stat, git --no-pager diff, または git show --stat HEAD で差分を取得
3. 変更ファイル・仕様書・日報・レビューメモで文脈を補足
4. git メタデータが読めない場合は現在のファイル内容から推測
```

この手順には根本的な問題があった。

**`git show HEAD` は最新の1コミットしか見ない。**

作業セッション中に複数のコミットが積み重なっていると、
HEAD 以外のコミットで変更されたファイルがエビデンスから抜け落ちる。
結果として、記事や日報が「今日やった作業の一部しか反映していない」状態になる。

---

## 解決策：6ステップのエビデンス収集プロセス

コミット `0be6876` で、ArticleWriterAgent と WorkSummaryAgent の両方に
体系的なエビデンス収集ルールが追加された。

### Step 1 — コミット範囲を把握する

```bash
# 直近30件のコミットを確認
git --no-pager log --oneline -30

# 今日のコミットを全件確認
git --no-pager log --oneline --since="$(date +%Y-%m-%d)"

# 現在のセッション（概ね今日）のコミットを確認
git --no-pager log --oneline --since="8 hours ago"
```

最も広い範囲を選び、最古のコミット SHA を `<base>`、最新を `HEAD` として記録する。

### Step 2 — 変更ファイルの全体像を把握する

```bash
# 変更ファイル一覧
git --no-pager diff --name-only <base>^ HEAD

# 行数サマリー付き
git --no-pager diff --stat <base>^ HEAD
```

**このリストにある全ファイルを読む。1ファイルの見落としが1記事トピックの見落としに直結する。**

### Step 3 — 全差分を読む

```bash
# 範囲全体の unified diff
git --no-pager diff <base>^ HEAD
```

差分が大きい場合はコミット単位で読む。

```bash
git --no-pager show --stat <sha>  # コミット単位のサマリー
git --no-pager show <sha>          # コミット単位の全差分
```

### Step 4 — リポジトリコンテキストで補足する

- `docs/design/` 配下の設計仕様書を読む
- `review/` 配下のレビューファイルを参照する
- 関連するテストファイルを読んで意図を把握する
- `blog/` を確認し、同じ変更について既に記事が存在しないかチェックする

### Step 5 — トピックを特定する

全コミットから見えた変更を列挙し、記事候補リストを作る。

- 新しいファイル・モジュール → 記事候補
- バグ修正 → 根本原因ごとに記事候補
- 設定・ルール・エージェント定義の変更 → 実質的な変更なら記事候補
- 単独で意味をなすリファクタリング → 記事候補

**2つの変更が独立して理解できる場合は、必ず別々のファイルに書く。**

### Step 6 — フォールバック

git メタデータが読めない場合は、現在のファイル内容とセッションコンテキストから推測し、
その旨を明記する。

---

## WorkSummaryAgent への適用

WorkSummaryAgent（日報を書くエージェント）にも同等の「Evidence Gathering Rules」セクションが追加された。
大きな違いは目的と出力形式。

| 観点 | ArticleWriterAgent | WorkSummaryAgent |
|---|---|---|
| 目的 | トピックごとに技術記事を書く | 今日の作業全体を日報にまとめる |
| Step 5 の出力 | 記事候補リスト（1トピック1ファイル） | 論理グループ別の作業項目 |
| `blog/` チェック | 重複記事を避けるため | 参照して日報に記載するため |

WorkSummaryAgent では「作業は raw なファイル変更ではなく、意図と成果で記述する」という原則が強調されている。

---

## この変更が解決する問題

### 問題: 「今日やった10個の変更のうち2個しか記事にならなかった」

以前: `git show HEAD` → 最新1コミットの変更だけが見える  
以後: `git log --since today` + `git diff <base>^ HEAD` → セッション全体が見える

### 問題: 「同じトピックが別の記事で重複した」

以後の Step 4 で `blog/` ディレクトリを確認するステップが義務化されたため、
既存記事との重複を事前にチェックできる。

### 問題: 「差分が大きすぎて全部読めなかった」

以後の Step 3 では「コミット単位で読む」フォールバックが明文化されているため、
大きな diff でも取りこぼしなく処理できる。

---

## まとめ

ArticleWriterAgent と WorkSummaryAgent に共通して追加された6ステップのエビデンス収集は、
「今日の変更を網羅的に把握する」という目標を達成するための手順の標準化だ。

`git show HEAD` 1コマンドでは見えなかった「セッション全体のコミット群」を  
体系的に走査することで、記事や日報の抜け漏れを構造的に防ぐ。  
トピックの洗い出し（Step 5）と重複チェック（Step 4の `blog/` 確認）を明示的に  
ステップとして組み込んだことで、エージェントの出力品質が安定する。
