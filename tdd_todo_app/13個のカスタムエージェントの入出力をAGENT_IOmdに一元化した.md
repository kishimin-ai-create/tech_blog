# 13 個のカスタムエージェントの入出力を AGENT_IO.md に一元化した

## 対象読者

- GitHub Copilot CLI のカスタムエージェント（`.agent.md`）を複数導入しているチーム
- エージェントを別プロジェクトへ移植することを考えている開発者
- 「どのエージェントに何を渡せばいいか」「出力ファイルはどこに出るか」を毎回調べているエンジニア

---

## 背景：13 個のエージェントが生み出す「どこに何がある」問題

このリポジトリには、TDD サイクルの各フェーズから記事執筆・PR 生成まで、13 個のカスタムエージェントが定義されている。

| 分類 | エージェント |
|---|---|
| TDD コアサイクル | OrchestratorAgent / RedAgent / GreenAgent / RefactorAgent |
| バグ修正 | FixAgent |
| コードレビュー | CodeReviewAgent / ReviewResponseAgent |
| テスト補完 | RegressionTestAgent |
| ドキュメント系 | ArticleWriterAgent / WorkSummaryAgent / PullRequestWriterAgent / OpenApiWriterAgent |
| UI | UIDesignAgent |

これだけの数になると、運用上いくつかの問題が表面化する。

- **入力の迷い**：「このエージェントに仕様書を渡すべきか？ブランチ名だけでいいのか？」
- **出力場所の把握��れ**：`review/`・`blog/`・`diary/`・`pull-request/`・`docs/spec/` など出力先がバラバラで、どこを見ればいいかわからない
- **移植コスト**：別プロジェクトへ持ち込む際、どのエージェントのどの行を書き換えればいいか把握するのが難しい

各エージェントの `.agent.md` を個別に読めば情報は得られるが、13 ファイルを横断して読むのは現実的ではない。

---

## 解決策：AGENT_IO.md という「玄関口」を作る

`commit 709bfeb` で `.github/AGENT_IO.md`（377 行）を新規作成した。

このファイルは各 `.agent.md` の**コピーではなく要約**であり、「エージェントを使う側・移植する側」が最初に見るべき単一の参照先として機能する。

---

## ファイルの構成

### 1. 早見表（ファイル冒頭）

ファイルの先頭に全エージェントを 1 テーブルにまとめた早見表がある。

```markdown
| エージェント           | 入力（何を渡すか）               | 出力ファイル・場所        |
|---|---|---|
| OrchestratorAgent  | 仕様書 `.md` のパス           | `review/`, `blog/`, `diary/` |
| RedAgent           | 仕様書 + ターゲット + スコープ    | テストファイル（`*.test.ts`） |
| ArticleWriterAgent | diff / 仕様書 / PR 文脈       | `blog/{title}.md`        |
| ...                | ...                          | ...                      |
```

「あのエージェント、何を渡すんだっけ？」という場面で、13 個の `.agent.md` を開かずにここだけ見れば済む。

### 2. エージェントごとの詳細セクション

早見表の下に、各エージェントの専用セクションが続く。各セクションは 3 つのサブテーブルで構成される。

#### ① 入力テーブル

何を渡すか・必須か任意かを明示する。

```markdown
### 入力
| 項目               | 内容                              |
|---|---|
| 仕様書（必須）      | `docs/spec/features/{feature}.md` などの Markdown パス |
| スコープ（任意）    | 対象レイヤー（Domain / Usecase / Interface / Component） |
```

#### ② 出力テーブル

どのファイルに何が生成されるかを示す。

```markdown
### 出力
| 成果物         | パス                                    |
|---|---|
| レビューファイル | `review/{work-description}-YYYYMMDD.md` |
```

#### ③ アプリごとに変わる設定

**ここが移植時の最重要セクション**。現プロジェクトに固有の値（コマンド・パス・フレームワーク名）を列挙する。

```markdown
### アプリごとに変わる設定
- テストフレームワーク（本プロジェクト: Vitest / React Testing Library）
- テスト実行コマンド（本プロジェクト: `npm run test`）
- テスト配置ディレクトリの慣習
```

### 3. プロジェクト共通設定マトリクス（ファイル末尾）

最後のセクションが「プロジェクト共通設定」で、全エージェントを横断した設定の一覧表だ。

```markdown
| 設定項目              | 本プロジェクトの値                      | 変更時の影響エージェント                         |
|---|---|---|
| バックエンドテストコマンド | `cd backend && npm run test`       | FixAgent / RefactorAgent / GreenAgent / RegressionTestAgent |
| 型チェックコマンド      | `cd backend && npm run typecheck`  | FixAgent / RefactorAgent                       |
| ブログ記事置き場        | `blog/`                            | ArticleWriterAgent                             |
| OpenAPI 仕様書         | `docs/spec/backend/openapi.yaml`   | OpenApiWriterAgent                             |
| PR テンプレート        | `.github/pull_request_template.md` | PullRequestWriterAgent                         |
```

「テストコマンドを変えたら何を直せばいい？」という疑問に、このテーブルが即答する。

## 移植ガイドとしての使い方

別プロジェクトへエージェント群を持ち込む際の手順を示す。

### ステップ 1：共通設定マトリクスから着手する

まず末尾のマトリクスを確認し、変更が必要な設定項目を洗い出す。テストコマンド・パス・フレームワーク名が現プロジェクトと異なる場合、**影響エージェント列**に列挙されたエージェントの `.agent.md` を修正する。

### ステップ 2：各エージェントの「アプリごとに変わる設定」を確認する

ステップ 1 で特定したエージェントの詳細セクションを開き、「アプリごとに変わる設定」の各項目を新プロジェクトの実態に合わせて書き換える。

### ステップ 3：出力パスが正しいか確認する

出力テーブルのパスが新プロジェクトのディレクトリ構成と一致しているかを確認する。`blog/` や `review/` が存在しない場合は事前に作成するか、パス設定を変更する。

---

## 設計上の判断

### なぜ各 .agent.md に情報を分散させたままにしなかったのか

各 `.agent.md` はエージェントの**振る舞いの定義**であり、長い。使う側がエージェントへの入力を確認するためだけに数百行を読むのは摩擦が大きい。`AGENT_IO.md` は「操作パネルの前面」として、実体（`.agent.md`）とは役割を分けている。

### なぜプロジェクト共通設定マトリクスをエージェント別セクションと分けたのか

テストコマンドのような横断的な設定は複数のエージェントに同じ影響を与える。エージェントごとにしか書かないと、一つの変更が漏れなく反映されたかどうかの確認が難しい。マトリクスで「影響範囲」を可視化することで、設定変更の抜け漏れを防ぐ。

## まとめ

| 課題 | AGENT_IO.md による解決 |
|---|---|
| 入力に迷う | 早見表で即座に確認できる |
| 出力場所がわからない | 出力テーブルで明示 |
| 移植時の修正漏れ | アプリごとの設定 + 共通マトリクスが起点になる |

エージェントの数が増えるほど、「何を渡すか・何が出るか・何を直すか」の情報を一か所に集約する価値は大きくなる。`.github/AGENT_IO.md` はその単一の参照先として機能し、新しいエージェントを追加するたびに対応するセクションを追記するだけで最新状態を維持できる。
