# コントローラー層統合テストで検証すること

## 対象読者

- Clean Architecture でコントローラー層のテストをどう書くか悩んでいる
- ユニットテストと統合テストで何を分担すべきか整理したい
- `deletedAt` などの内部フィールドをレスポンスから除く設計をテストしたい

---

## コントローラーの役割

このプロジェクトのコントローラーは「薄いアダプター」として設計されている。

```typescript
export function createAppController(appUsecase: AppUsecase): AppController {
  async function create(body: unknown): Promise<JsonHttpResponse> {
    try {
      const app = await appUsecase.create(parseCreateAppInput(body));
      return presentSuccess(presentApp(app), 201);
    } catch (error) {
      return handleControllerError(error);
    }
  }
  // ...
}
```

コントローラーが担うのは以下の3つだ。

1. **リクエストのパース・バリデーション**（`parseCreateAppInput`）
2. **ユースケースの呼び出し**（`appUsecase.create`）
3. **レスポンスへの変換**（`presentSuccess(presentApp(app), 201)`）

ビジネスロジックはユースケース層に委ねる。コントローラーは HTTP と ユースケースを「つなぐ」層だ。

---

## ユニットテストとの違い

コントローラーのユニットテストでは `appUsecase` を `vi.fn()` でモックする。このとき「バリデーション → ユースケース呼び出し → レスポンス変換」の連携は確認できない。

統合テストでは、実際の `InMemoryRepository` と実際の `AppInteractor` を使う。

```typescript
function setup() {
  const storage = createInMemoryStorage();
  const appRepository = createInMemoryAppRepository(storage);
  const todoRepository = createInMemoryTodoRepository(storage);
  const interactor = createAppInteractor({ appRepository, todoRepository });
  const controller = createAppController(interactor);
  return { controller, interactor };
}
```

「バリデーションエラーが 422 になること」「存在しない ID が 404 になること」といった HTTP ステータスコードの最終確認は、ここで行う。

## 検証項目1：HTTP ステータスコードの正確さ

```typescript
describe('create', () => {
  it('returns 201 with success:true and app DTO on valid input', async () => {
    const res = await ctx.controller.create({ name: 'My App' });
    expect(res.status).toBe(201);
    expect(res.body.success).toBe(true);
  });

  it('returns 422 when name is missing', async () => {
    const res = await ctx.controller.create({});
    expect(res.status).toBe(422);
    expect(res.body.error?.code).toBe('VALIDATION_ERROR');
  });

  it('returns 409 when the name is already taken', async () => {
    await ctx.controller.create({ name: 'Dup' });
    const res = await ctx.controller.create({ name: 'Dup' });
    expect(res.status).toBe(409);
    expect(res.body.error?.code).toBe('CONFLICT');
  });
});
```

特に注目すべきは「重複チェック」のシナリオだ。最初の `create` で実際にデータが保存され、2回目の `create` で CONFLICT が発生する。この連鎖はモックでは再現しにくい。

## 検証項目2：`deletedAt` の除外

エンティティの `deletedAt` フィールドはクライアントへのレスポンスに含めない仕様だ。

```typescript
it('DTO does not include deletedAt', async () => {
  const res = await ctx.controller.create({ name: 'App' });
  expect(
    (res.body.data as Record<string, unknown>)
  ).not.toHaveProperty('deletedAt');
});
```

`presentApp()` の実装でこれを担保している。

```typescript
export function presentApp(app: AppEntity) {
  return {
    id: app.id,
    name: app.name,
    createdAt: app.createdAt,
    updatedAt: app.updatedAt,
    // deletedAt は含めない
  };
}
```

スプレッド演算子（`{ ...app }`）ではなく、必要なフィールドだけを列挙する構造にすることで、エンティティに新しいフィールドが追加されてもクライアントに自動で漏れない。

「`deletedAt` がレスポンスに含まれないこと」は仕様上の重要な制約なので、明示的なテストケースとして存在する価値がある。

## 検証項目3：エラーチェーンの完全な確認

```typescript
describe('get', () => {
  it('returns 404 for an unknown id', async () => {
    const res = await ctx.controller.get(GHOST_ID);
    expect(res.status).toBe(404);
    expect(res.body.error?.code).toBe('NOT_FOUND');
  });
});
```

`GHOST_ID` は実際には存在しない UUID だ。コントローラー → インタラクター → リポジトリ → null 返却 → `NOT_FOUND` エラースロー → `handleControllerError` → `presentError` → 404 という一連の流れを、モックなしで端から端まで確認している。

## 検証項目4：ボディが `null` の場合

```typescript
it('returns 422 when body is null', async () => {
  const res = await ctx.controller.create(null);
  expect(res.status).toBe(422);
});
```

HTTP リクエストからパースされたボディは `null` になる可能性がある（JSON が `null` 単体だった場合など）。`parseCreateAppInput(null)` がエラーを投げ、422 を返すことを確認する。

`toRecord()` の実装でこれを処理している。

```typescript
function toRecord(value: unknown): Record<string, unknown> {
  if (typeof value !== 'object' || value === null || Array.isArray(value)) {
    return {};  // ← null や配列は空オブジェクトとして扱う
  }
  return value as Record<string, unknown>;
}
```

`null` は `typeof null === 'object'` という JavaScript の仕様があるため、別途 `value === null` チェックが必要だ。

## テスト間の隔離

```typescript
describe('AppController integration', () => {
  let ctx: ReturnType<typeof setup>;

  beforeEach(() => {
    ctx = setup();  // ← テストごとに新しいストレージ・インタラクター・コントローラー
  });
  // ...
});
```

`beforeEach` で `setup()` を呼ぶことで、各テストが独立した状態から始まる。前のテストが作成したデータが次のテストに影響しない。

---

## まとめ

- コントローラー統合テストは「バリデーション → ユースケース → レスポンス変換」の連携を実際の実装で確認する
- ステータスコード（201, 422, 409, 404）を各シナリオで明示的にアサートする
- `deletedAt` の除外は `not.toHaveProperty()` で明確にテストする
- `null` ボディや重複データなど、実際のリクエストで発生しうるエッジケースを網羅する
- `beforeEach` で `setup()` を呼ぶことでテスト間の隔離を保証する
