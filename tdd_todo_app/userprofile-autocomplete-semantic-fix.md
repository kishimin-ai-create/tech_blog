# UserProfilePage の autoComplete 属性を適切なセマンティック値に修正した

## エラー概要

`frontend/src/features/auth/pages/UserProfilePage.tsx` のパスワード・メールフィールドに `autoComplete="off"` が設定されていた。これはブラウザのパスワードマネージャーとの連携を完全に遮断する設定であり、プロフィール編集フォームには適切でなかった。コミット `da7fe67` で正しいセマンティック値に置き換えた。

---

## 原因

### 変更前のコード

```tsx
// 変更前 — 3 フィールドすべてが autoComplete="off"
<input type="text"     autoComplete="off" />  {/* メールアドレス */}
<input type="password" autoComplete="off" />  {/* 新しいパスワード */}
<input type="password" autoComplete="off" />  {/* 現在のパスワード */}
```

`autoComplete="off"` は「ブラウザはこのフィールドへの自動補完を行わないでほしい」というシグナルである。ログイン画面のように「フォームを開くたびに空にしたい」場面では有効だが、プロフィール編集フォームにこれを使うと次の問題が生じる。

- **パスワードマネージャーがパスワード変更を検知しない** — `new-password` を指定しないと、ブラウザは新しいパスワードの保存・更新を促さない
- **メールフィールドへのオートフィルが無効になる** — `email` を指定することでブラウザが保存済みのアドレスを提案できる
- **アクセシビリティ支援ツールへのヒントが失われる** — `autoComplete` の値はスクリーンリーダーやフォームオートフィルツールが入力の種類を判断する手がかりになる

### なぜ `"off"` が最初に使われたか

ログインフォームやサインアップフォームでの自動入力トラブル（保存済み認証情報がページ遷移後も残る問題）を解消するため、このプロジェクトではいくつかのフィールドに `autoComplete="off"` が適用されてきた経緯がある（参照: `20260510-fix-login-autofill.md`）。  
その際の設定が `UserProfilePage` にも踏襲され、プロフィール編集という異なるコンテキストにも `"off"` が付いてしまっていたと考えられる。

---

## 修正内容

### 変更後のコード

```tsx
// 変更後 — フィールドの用途に合った autoComplete 値
<input type="text"     autoComplete="email"            />  {/* メールアドレス */}
<input type="password" autoComplete="new-password"     />  {/* 新しいパスワード */}
<input type="password" autoComplete="current-password" />  {/* 現在のパスワード */}
```

| フィールド | 値 | 意味 |
|---|---|---|
| メールアドレス | `"email"` | ブラウザが保存済みのメールアドレスを提案できる |
| 新しいパスワード | `"new-password"` | ブラウザが強力なパスワード生成を提案し、保存を促す |
| 現在のパスワード | `"current-password"` | パスワードマネージャーが保存済みパスワードを補完できる |

### `"new-password"` と `"current-password"` の使い分け

パスワード変更フォームでこの 2 つを正しく使い分けることは重要である。

- `"current-password"` — 「現在のパスワードを入力してください」欄に使う。パスワードマネージャーが保存済みの認証情報を補完する
- `"new-password"` — 「新しいパスワードを入力してください」欄に使う。ブラウザはここに保存済みパスワードを補完せず、代わりに強力なランダムパスワードの生成を提案する

2 つのフィールドに同じ `"current-password"` を設定してしまうと、ブラウザは新しいパスワードに現在の認証情報を補完しようとするため、意図しない入力が起こりうる。

---

## ログインフォームとの設定の違い

このプロジェクトには `autoComplete` に関連する修正が複数存在し、フォームの用途ごとに意図的に異なる値が使われている。

| フォーム | 設定値 | 理由 |
|---|---|---|
| `LoginPage`（メール） | `"off"` | ページ遷移後もフィールドを空にしたい |
| `LoginPage`（パスワード） | `"off"` | 同上 |
| `UserProfilePage`（メール） | `"email"` | パスワードマネージャーとの連携が望ましい |
| `UserProfilePage`（新パスワード） | `"new-password"` | ブラウザによるパスワード更新提案を有効にする |
| `UserProfilePage`（現パスワード） | `"current-password"` | 保存済みパスワードの補完を許可する |

**フォームの目的が違えば `autoComplete` の設定も変わる。** ログインフォームは「毎回クリーンな状態で開く」UX を優先しているが、プロフィール編集フォームは「パスワードマネージャーとの連携」を優先すべきコンテキストである。

---

## まとめ

| 項目 | 内容 |
|---|---|
| **問題** | プロフィール編集フォームの全フィールドに `autoComplete="off"` が設定されていた |
| **影響** | パスワードマネージャーによるパスワード更新提案が無効になる |
| **修正** | `email` / `new-password` / `current-password` という用途に合ったセマンティック値に変更 |
| **変更ファイル** | `frontend/src/features/auth/pages/UserProfilePage.tsx` |

`autoComplete` は小さな属性だが、パスワードマネージャーとの連携やブラウザの補完動作を直接制御する。フォームを実装するときは「このフォームはユーザーに何をさせるものか」を考えて値を選ぶことが大切である。ログインフォームとパスワード変更フォームを同じ設定で扱わない。
