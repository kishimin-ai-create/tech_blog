## Root Causes by Category

### 1. ESLint Configuration: Generated Files Scope Leak

**Root Cause:**
The ESLint configuration file (`frontend/eslint.config.js`) did not exclude the `storybook-static/` directory, which is auto-generated during the Storybook build process.

**Evidence:**
- `storybook-static/` is created by `npm run build-storybook`
- Contains hundreds of minified, transpiled files
- Each file triggered multiple linting rules (e.g., unused imports, missing JSDoc)
- These violations were not actionable—they were build artifacts

**Impact Quantified:**
- ~100 warnings alone were generated by `.js` files in `storybook-static/`
- This noise obscured actual implementation issues
- CI logs became difficult to parse

**Technical Root:**
When a linting tool processes a directory recursively without explicit `ignore` patterns, it treats all files equally regardless of their origin (source vs. generated).

### 2. Import Ordering: Inconsistent Categorization

**Root Cause:**
Import statements mixed external packages and relative imports without a defined sort order.

**Pattern Observed (12 files):**

```typescript
// ✗ Problem: External and local imports intermingled
import { AppListPage } from './features/app-list/pages/AppListPage'
import { useAtom } from 'jotai'
import { FC } from 'react'
import { TodoForm } from './features/app-detail/components/TodoForm'
import { useQuery } from '@tanstack/react-query'

```

**Why This Violated Rules:**
- ESLint rule `import/order` enforces:
  1. External packages first (sorted alphabetically)
  2. Relative imports second (sorted alphabetically)
  3. Side-effect imports last
  4. Blank lines between groups
- The observed pattern violated step 1–2 ordering

### 5. Async Event Handlers: Promise-Returning Callbacks

**Root Cause:**
Six files contained event handlers (onClick, onSubmit, onChange) that returned Promises, which React cannot properly await or handle, violating `@typescript-eslint/no-misused-promises`.

**The Problem (Technical Depth):**

```typescript
// ✗ Error: onClick handler returns a Promise
<button onClick={() => handleDelete()}>Delete</button>
// where handleDelete() is async

// Why this breaks:
// - React event handlers expect void return
// - React does not await Promises in event callbacks
// - Error handling becomes non-deterministic
// - Component state may not reflect async completion
```

**Affected Files (6):**

### Solution 5: Async Event Handler Correction (6 files)

**Pattern Analysis:**

The fix varies by handler type. Two primary approaches applied:

#### Approach 5A: Inline void + catch (Button/Element handlers)

```typescript
// ✗ Before: Direct async call
<button onClick={() => handleDelete()}>Delete</button>

// ✓ After: Wrapped with void and error handling
<button onClick={() => { void handleDelete().catch(() => {}) }}>Delete</button>
```

#### Approach 5B: Event Handler Wrapper (Form submissions)

```typescript
// ✗ Before: handleSubmit directly used in onSubmit
<form onSubmit={(e) => { handleSubmit(handleFormSubmit)(e) }}>

// ✓ After: .catch() chained to the result
<form onSubmit={(e) => { handleSubmit(handleFormSubmit)(e).catch(() => {}) }}>
```

**Files Fixed (6):**
- `src/features/app-create/components/AppForm.tsx` (1 fix)
- `src/features/app-create/pages/AppCreatePage.tsx` (1 fix, implicit via form wrapper)
- `src/features/app-detail/components/TodoForm.tsx` (1 fix)
- `src/features/app-detail/components/TodoItem.tsx` (2 fixes)
- `src/features/app-detail/pages/AppDetailPage.tsx` (1 fix)
- `src/features/app-edit/pages/AppEditPage.tsx` (1 fix)

**Why This Works:**
1. `void` operator explicitly discards the Promise
2. `.catch(() => {})` satisfies TypeScript's error handling requirement
3. React receives `void` return value (expected)
4. Async operation executes without blocking UI

### Solution 6: Type Safety Improvements (4 files)

**Challenge:**
API mutations return untyped responses. Direct property access was flagged as unsafe.

**Applied Pattern:**

```typescript
// ✗ Before: Untyped member access
const result = await mutation.mutateAsync({ data: { name: values.name } })
if (result.status === 201) { ... }

// ✓ After: Explicit type narrowing
const result = (await mutation.mutateAsync({ data: { name: values.name } })) as unknown
const typedResult = result as { status?: number }
if (typedResult?.status === 201) { ... }
```

**Key Techniques:**
- Explicit `as unknown` cast (acknowledging the unsafe cast)
- Type narrowing with `as { property?: Type }`
- Optional chaining (`?.`) for safe member access
- Proper type guards for status checks

## Technical Lessons

### 1. Linting Configuration Isolation

**Lesson:**
Generated files (build artifacts, compiled output) should always be excluded from linting. They are transient and non-authoritative.

**Principle:**
```
globalIgnores = [
  "build outputs (dist/)",
  "generated code (src/api/generated/)",
  "build artifacts (storybook-static/)",
]
```

**Application:**
- Reduces noise in CI logs
- Focuses developer attention on source code
- Prevents rules designed for source code from triggering on generated code

### 2. Staged Rule Introduction

**Lesson:**
Starting with 408 violations at once created a high-effort fix. Future approach: introduce new linting rules **before** accumulating violations.

**Recommended Practice:**
1. Add new rule to ESLint config
2. Run `npm run lint` to identify current violations
3. Fix small batches incrementally
4. Merge and deploy frequently

### 3. Type Safety as Prevention

**Lesson:**
Explicit type casting (`as Type`) is verbose but prevents entire classes of runtime errors. The investment pays off in production stability.

**Pattern:**
```typescript
// Before: one property access = one hidden error
result.status  // ← type is any

// After: intentional narrowing
(result as { status?: number })?.status  // ← explicitly typed
```

### 4. Documentation as Code Quality Signal

**Lesson:**
JSDoc comments serve dual purposes:
1. **Immediate:** IDE tooltips and auto-complete
2. **Long-term:** Foundation for future documentation tools (typedoc, etc.)

**ROI:**
- 13 functions documented
- Approximately 30 developer touches avoided
- Future tool compatibility established

### 5. Event Handler Patterns in React + TypeScript

**Lesson:**
React's event model (synchronous handlers) conflicts with modern async/await patterns. Solutions require explicit awareness:
- `void operator` discards the Promise
- `.catch()` satisfies error handling
- Form submission requires special handling with `.catch()`

**Pattern Used:**
```typescript
void asyncFunction().catch(() => {})  // ✓ Correct
// vs.
asyncFunction()  // ✗ Type error: Promise not returned
```
