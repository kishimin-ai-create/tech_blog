# マルチエージェントTDDでオーケストレーターが自分でコードを書かないようにした

## 対象読者

- GitHub Copilot CLI などでマルチエージェント運用を設計している人
- TDD のオーケストレーターにどこまで権限を持たせるべきか悩んでいる人

## この記事の範囲

この記事では、`.github/agents/OrchestratorAgent.agent.md` と `diary/20260425.md` を主な根拠に、**OrchestratorAgent がサブエージェント失敗時に自分でコードを書いてしまう問題**への対処を整理します。  
backend の Todo API 実装そのものは扱いません。

> 注: この実行環境では `git status` や直近 diff の出力を直接取得できなかったため、現行ファイルと最新の作業記録を基準に直近変更を再構成しています。

## 背景

2026-04-25 の作業記録では、Red Agent や Green Agent が必要なファイルを作れなかったときに、OrchestratorAgent が一般用途のエージェント経由で**自分でテストや実装を書いてしまう**事象が記録されていました。

これは一見すると復旧策ですが、実際には次の問題があります。

- 役割分離が壊れる
- 失敗したサブエージェントの問題が見えなくなる
- 「誰が何を保証するか」が曖昧になる

## 原因

原因は、オーケストレーターに「失敗時の停止条件」が十分に書かれていなかったことです。  
以前は Red → Green → Refactor → Review の流れ自体は定義されていても、**サブエージェントが成果物を出せなかったときに、どこで止まるべきか**が弱かったと読めます。

## 対応

`OrchestratorAgent.agent.md` では、失敗時の扱いが明確化されました。

```md
❌ Generate code yourself — ABSOLUTE.
❌ Fall back to writing code when a sub-agent fails
```

あわせて、各フェーズに具体的な失敗時ルールが入りました。

- Red / Green フェーズ: 1回だけリトライし、それでも失敗したら停止して報告
- Refactor フェーズ: 変更なしも正当な結果として受け入れる
- Review フェーズ: レビューファイルの存在を完了条件に含める

現行ファイルでも、次のように確認できます。

- `.github/agents/OrchestratorAgent.agent.md`
  - 「自分でコードを書かない」
  - 「1回だけ再試行し、だめなら止まる」
  - 「Review 完了前に done にしない」

## もう1つ入ったルール: 確認待ちをしない

同日の作業記録では、全エージェントに「進めてよいですか」と確認しないルールも追加されています。  
現行の `.github/instructions/copilot-instructions.md` には次があり、`.github/agents/ArticleWriterAgent.agent.md` や `.github/agents/GreenAgent.agent.md` にも同趣旨の禁止が確認できます。

- 指示を受けたら即実行する
- approval 型の質問で止まらない
- ただし、安全に進められないほど要件が足りないときだけ必要な事実を聞く

ここで重要なのは、**自律的に進むこと**と**役割を越えて代行すること**は別だという点です。

- 自律性: 許可待ちで止まらない
- 境界管理: 失敗した他エージェントの仕事を勝手に肩代わりしない

今回の修正は、この2つを同時に整理しています。

## 実装上のポイント

実際のルール設計として有効だったのは、抽象論ではなくフェーズごとの動作を書いたことです。

1. 禁止事項を強い文言で固定する
2. リトライ回数を明示する
3. 「変更なし」や「停止して報告」を正常系として定義する
4. レビューファイルのような外部成果物を完了条件に入れる

オーケストレーター設計では、成功フローよりも**失敗フローをどう終わらせるか**のほうが重要だと分かります。

## 注意点

- 現在の確認結果は主に現行ファイルと `diary/20260425.md` ベースです
- 全エージェントの差分までは直接比較できていないため、記事中では確認できたファイル名を明示しました
- ルールを強くしても、サブエージェントの失敗原因そのものが解決するわけではありません

## まとめ

今回の変更の価値は、オーケストレーターを「何でも埋める便利役」から、**順序・責務・停止条件を守る指揮者**に戻したことです。

マルチエージェント構成では、失敗時に賢く見せることよりも、

- どこで止まるか
- 誰が何を書いてよいか
- 何をもって完了とするか

を明文化するほうが、結果として運用しやすくなります。
