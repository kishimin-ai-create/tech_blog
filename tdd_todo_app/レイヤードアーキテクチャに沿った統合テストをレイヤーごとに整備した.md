# レイヤードアーキテクチャに沿った統合テストをレイヤーごとに整備した

## 対象読者

- Layered Architecture（レイヤードアーキテクチャ）で TypeScript バックエンドを書いている
- ユニットテストは書けているが、統合テストをどう設計するか悩んでいる
- 実際のテストコードを見ながら設計の考え方を掴みたい

## この記事が扱う範囲

- `backend/src/tests/integrations/` 配下のフォルダ構成と各レイヤーのテスト設計
- `createInMemoryStorage()` を共有する「Shared Storage パターン」
- インターフェース契約テストの実装方法
- ユニットテストとの役割分担

CI ワークフローの変更については別記事で扱う。

---

## 背景：統合テストが1つのフォルダに集中していた

このプロジェクトは Hono + TypeScript で構築した Todo API で、アーキテクチャは次のレイヤー構成を取っている。

```
backend/src/
├── models/        # AppEntity, TodoEntity, AppError
├── repositories/  # AppRepository, TodoRepository（インターフェース）
├── services/      # AppInteractor, TodoInteractor（ユースケース実装）
├── controllers/   # AppController, TodoController, HttpPresenter, RequestValidation
└── infrastructure/# HonoApp, InMemoryRepositories, InMemoryStorage, Registry
```

改修前は `tests/integrations/` 直下に `app.test.ts` と `index.test.ts` という2ファイルしかなく、HTTP エンドポイントレベルの確認に留まっていた。ユニットテスト（`vi.fn()` によるモック）はソースファイルと同階層に co-locate されているが、複数レイヤーを跨いだ結合の検証は手薄だった。

今回の整備で `tests/integrations/` 配下をアーキテクチャのレイヤー構成と対応させ、17 ファイルの統合テストスイートを構築した。

## 設計の核心：Shared Storage パターン

統合テストで最も重要な設計判断は **`createInMemoryStorage()` を app リポジトリと todo リポジトリで共有する** という点だ。

```typescript
function setup() {
  const storage = createInMemoryStorage();  // ← 1つの storage を作る
  const appRepository  = createInMemoryAppRepository(storage);   // ← 同じ storage を渡す
  const todoRepository = createInMemoryTodoRepository(storage);  // ← 同じ storage を渡す
  const interactor: AppUsecase = createAppInteractor({ appRepository, todoRepository });
  return { storage, appRepository, todoRepository, interactor };
}
```

なぜこれが重要か。`AppInteractor.delete()` はアプリを論理削除するとき、紐づく Todo も cascade delete する仕様だ。この処理は `todoRepository` を通じて行われるが、**storage が別インスタンスだと cascade が無声で失敗する**。

実際のテストで確認している内容：

```typescript
it('cascade-deletes associated todos via shared repository', async () => {
  const app = await ctx.interactor.create({ name: 'Cascade' });
  const todo: TodoEntity = {
    id: 'todo-cascade',
    appId: app.id,
    title: 'Will be deleted',
    completed: false,
    createdAt: '2024-01-01T00:00:00.000Z',
    updatedAt: '2024-01-01T00:00:00.000Z',
    deletedAt: null,
  };
  await (ctx.todoRepository as TodoRepository).save(todo);
  await ctx.interactor.delete({ appId: app.id });
  const foundTodo = await ctx.todoRepository.findActiveById(app.id, 'todo-cascade');
  expect(foundTodo).toBeNull();  // ← cascade が正しく動いた
});
```

ユニットテストは `vi.fn()` でリポジトリをモックするため、このようなクロスリポジトリのシナリオは検証できない。統合テストが必要な理由がここにある。

## インターフェース契約テスト

`services/app-usecase.test.ts` には少し変わったテストがある。

```typescript
function makeUsecase(): AppUsecase {
  const storage = createInMemoryStorage();
  return createAppInteractor({
    appRepository: createInMemoryAppRepository(storage),
    todoRepository: createInMemoryTodoRepository(storage),
  });
}

describe('AppUsecase interface contract', () => {
  it('createAppInteractor satisfies the AppUsecase interface', () => {
    const usecase: AppUsecase = makeUsecase();
    // ↑ 型代入が通ることでコンパイル時にインターフェース適合を保証
    expect(typeof usecase.create).toBe('function');
    expect(typeof usecase.list).toBe('function');
    expect(typeof usecase.get).toBe('function');
    expect(typeof usecase.update).toBe('function');
    expect(typeof usecase.delete).toBe('function');
  });
```

`const usecase: AppUsecase = createAppInteractor(...)` という型注釈が鍵だ。`createAppInteractor` が返す型が `AppUsecase` を満たしていなければ、TypeScript のコンパイル時点でエラーになる。実行時テストと型レベルの検証を同時に行っている。

## レイヤーごとのテスト設計

### repositories 層：契約テスト

`repositories/app-repository.test.ts` は **リポジトリインターフェースの契約** を検証する。実装クラスに直接依存せず、インターフェース型で受け取ることで「どの実装を渡しても同じ契約を満たせばよい」という設計を担保する。

```typescript
describe('AppRepository contract', () => {
  let repo: AppRepository;  // ← インターフェース型

  beforeEach(() => {
    repo = createInMemoryAppRepository(createInMemoryStorage());
  });

  it('returned entity is a defensive copy, not the original reference', async () => {
    const app = makeApp('app-1', 'Copy Test');
    await repo.save(app);
    const found = await repo.findActiveById('app-1');
    expect(found).not.toBe(app);  // ← 参照の独立性を検証
  });
```

「保存したオブジェクトへの参照を返さない」という防御的コピーの確認など、インターフェースの文書には書きにくい暗黙の契約もテストで明示できる。

### services 層：ビジネスロジックの検証

`services/app-interactor.test.ts` では create / list / get / update / delete の全操作に加え、エラーシナリオを網羅する。

注目すべきケース：

```typescript
it('allows updating to the same name (self-exclusion)', async () => {
  const app = await ctx.interactor.create({ name: 'Same' });
  const updated = await ctx.interactor.update({ appId: app.id, name: 'Same' });
  expect(updated.name).toBe('Same');
  // ← 自分自身の名前への更新は CONFLICT にならない
});
```

「他のアプリと同名は CONFLICT だが、自分自身の名前へ更新するのは許可する」という仕様は、ユニットテストではモックの設定で表現しにくい。実際のリポジトリを使う統合テストで確認するのが適切だ。

### controllers 層：レスポンス shape の検証

`controllers/app-controller.test.ts` では HTTP レスポンスとしての DTO 変換を検証する。特に「`deletedAt` が DTO に含まれないこと」は要件として明示的にテストされている。

```typescript
it('DTO does not include deletedAt', async () => {
  const res = await ctx.controller.create({ name: 'App' });
  expect((res.body.data as Record<string, unknown>)).not.toHaveProperty('deletedAt');
});
```

エンティティの内部フィールドをクライアントに漏らさないことは、API 設計の重要な契約だ。

`controllers/request-validation.test.ts` では境界値を含むエッジケースを検証する。

```typescript
it('throws VALIDATION_ERROR when name exceeds 100 characters', () => {
  expect(() => parseCreateAppInput({ name: 'a'.repeat(101) })).toThrow(
    expect.objectContaining({ code: 'VALIDATION_ERROR' }),
  );
});

it('accepts a name of exactly 100 characters', () => {
  const result = parseCreateAppInput({ name: 'a'.repeat(100) });
  expect(result.name).toHaveLength(100);
});
```

100文字と101文字の境界を明示的に確認することで、バリデーションルールの曖昧さを排除している。

### infrastructure 層：結線の確認

`infrastructure/registry.test.ts` は DI コンテナ（Registry）の結線を確認するスモークテストだ。

```typescript
it('each call to createBackendRegistry creates an isolated registry', async () => {
  const r1 = createBackendRegistry();
  const r2 = createBackendRegistry();
  await r1.app.request('http://localhost/api/v1/apps', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ name: 'R1 App' }),
  });
  const res = await r2.app.request('http://localhost/api/v1/apps');
  const json = await res.json() as { data: unknown[] };
  expect(json.data.length).toBe(0);  // ← r1 のデータが r2 に漏れない
});
```

テスト間でデータが漏れないこと（テスト隔離）は、統合テストを信頼できるものにするための前提条件だ。各テストケースで `beforeEach` に `setup()` を呼ぶか、独立した Registry を生成することで隔離を保っている。

## ユニットテストとの役割分担

| 観点 | ユニットテスト | 統合テスト |
|---|---|---|
| 依存の扱い | `vi.fn()` でモック | 実装クラスをそのまま使用 |
| 検証対象 | 単一クラスのロジック | 複数レイヤーを跨いだ結合 |
| 実行速度 | 高速 | やや低速 |
| テストの場所 | ソースファイルと同階層（co-locate） | `tests/integrations/` 配下 |
| 得意なシナリオ | 条件分岐・エラーハンドリング | cascade delete・DTO変換・契約適合 |

ユニットテストで「Service が Repository の `findActiveById` を呼ぶか」は確認できる。しかし「実際に削除済みエンティティが返ってこないか」は統合テストでないと確認できない。両者は補完関係にある。

---

## まとめ

- **フォルダ構成をアーキテクチャと揃える**ことで、テストの目的と場所が一致する
- **Shared Storage パターン**を使うことで、cascade delete などのクロスリポジトリ動作を正しく検証できる
- **インターフェース型注釈による契約テスト**で、実装がインターフェースを満たすことをコンパイル時にも保証する
- 統合テストの価値は「実装をモックせず、複数レイヤーを跨いだ実際の動作を確認すること」にある
- ユニットテストと統合テストは競合するものではなく、異なる粒度で同じコードを守る補完関係にある
