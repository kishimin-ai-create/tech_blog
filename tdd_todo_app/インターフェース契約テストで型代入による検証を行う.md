# インターフェース契約テスト：型代入でビジネスロジック実装の正確性を検証する

## 対象読者

- TypeScript で usecase / interactor パターンを使っている
- インターフェース定義と実装が正しく一致することを確認したい
- 型システムでテスト可能性を高める方法を知りたい

## この記事が扱う範囲

- `AppUsecase` / `TodoUsecase` インターフェース定義
- 型代入テストによる contract verification
- メソッド名・パラメータ型・戻り値型の一貫性検証
- 実装が interface を満たすことの自動検証

---

## 背景：interface と implementation の乖離

TypeScript で interface を定義し、その実装（Interactor）を書く場合、以下のリスクがある：

- **メソッド名の誤り** — `creat()` と `create()` の typo
- **パラメータ型の違い** — Input 型が一致していない
- **戻り値型の違い** — Promise<AppEntity> が Promise<void> になっていないか
- **メソッド数の不足** — interface の一部メソッドが実装されていない

これらを**コンパイル時に検出**するのが「型代入テスト」である。

---

## AppUsecase インターフェース定義

```typescript
export type CreateAppInput = { name: string };
export type GetAppInput = { appId: string };
export type UpdateAppInput = { appId: string; name?: string };
export type DeleteAppInput = { appId: string };

export interface AppUsecase {
  create(input: CreateAppInput): Promise<AppEntity>;
  list(): Promise<AppEntity[]>;
  get(input: GetAppInput): Promise<AppEntity>;
  update(input: UpdateAppInput): Promise<AppEntity>;
  delete(input: DeleteAppInput): Promise<void>;
}
```

**設計のポイント：**

1. **Input 型の明示化** — 各操作に固有の input type を定義
2. **戻り値の統一** — create / get / update は `Promise<AppEntity>` / delete は `Promise<void>`
3. **パラメータの型安全性** — optional フィールド（`name?`）を許可

## 型代入テスト（type assignment test）

### AppUsecase の型代入テスト

```typescript
describe('AppUsecase interface contract', () => {
  function makeUsecase(): AppUsecase {
    const storage = createInMemoryStorage();
    return createAppInteractor({
      appRepository: createInMemoryAppRepository(storage),
      todoRepository: createInMemoryTodoRepository(storage),
    });
  }

  it('createAppInteractor satisfies the AppUsecase interface', () => {
    // 型代入：interactor が AppUsecase を満たすことをコンパイル時検証
    const usecase: AppUsecase = makeUsecase();
    expect(usecase).toBeDefined();
  });

  it('create resolves to an AppEntity shape', async () => {
    const usecase = makeUsecase();
    const app = await usecase.create({ name: 'Contract Test' });
    expect(app).toMatchObject({
      id: expect.any(String),
      name: 'Contract Test',
      createdAt: expect.any(String),
      updatedAt: expect.any(String),
      deletedAt: null,
    });
  });

  it('list resolves to an array', async () => {
    const usecase = makeUsecase();
    const result = await usecase.list();
    expect(Array.isArray(result)).toBe(true);
  });

  it('get resolves to the app by id', async () => {
    const usecase = makeUsecase();
    const created = await usecase.create({ name: 'Get Contract' });
    const found = await usecase.get({ appId: created.id });
    expect(found.id).toBe(created.id);
  });

  it('update resolves to the modified app', async () => {
    const usecase = makeUsecase();
    const created = await usecase.create({ name: 'Original' });
    const updated = await usecase.update({ appId: created.id, name: 'Updated' });
    expect(updated.name).toBe('Updated');
  });

  it('delete resolves to void', async () => {
    const usecase = makeUsecase();
    const created = await usecase.create({ name: 'Delete Test' });
    const result = await usecase.delete({ appId: created.id });
    expect(result).toBeUndefined();
  });
});
```

**型代入テストの仕組み：**

```typescript
const usecase: AppUsecase = makeUsecase();
//       ^^^^^^^^^^^^^^^^
// このアサーションでコンパイルエラーが発生する場合：
// - makeUsecase() の戻り値が AppUsecase を実装していない
// - メソッド名が異なる
// - パラメータ型が異なる
// - 戻り値型が異なる
```

## 型代入テストで検出可能なエラー

### パターン1：メソッド名の誤り

```typescript
// ❌ 実装が間違っていた場合
export function createAppInteractor(...): AppUsecase {
  return {
    creat: async (input) => { /* ... */ }, // typo: create ではなく creat
    list: async () => { /* ... */ },
    // ...
  };
}

// コンパイルエラー：
// Type '{ creat: ...; list: ... }' is not assignable to type 'AppUsecase'
//   Object literal may only specify known properties, and 'creat' does not exist in type 'AppUsecase'
```

### パターン2：戻り値型の違い

```typescript
// ❌ 戻り値が void だが interface では Promise<AppEntity> を期待
export function createAppInteractor(...): AppUsecase {
  return {
    create: async (input) => { /* 何も return しない */ },
    // ...
  };
}

// コンパイルエラー：
// Type 'Promise<undefined>' is not assignable to type 'Promise<AppEntity>'
```

### パターン3：パラメータ型の違い

```typescript
// ❌ input 型が一致していない
export function createAppInteractor(...): AppUsecase {
  return {
    create: (input: { appName: string }) => { /* ... */ },
    // interface では CreateAppInput = { name: string }
  };
}

// コンパイルエラー：
// Type of 'create' is not compatible with 'AppUsecase'
//   Parameter 'input' has a different type from interface 'AppUsecase'
```

## TodoUsecase の型代入テスト

```typescript
describe('TodoUsecase interface contract', () => {
  function makeUsecase(): TodoUsecase {
    const storage = createInMemoryStorage();
    return createTodoInteractor({
      appRepository: createInMemoryAppRepository(storage),
      todoRepository: createInMemoryTodoRepository(storage),
    });
  }

  it('createTodoInteractor satisfies TodoUsecase interface', () => {
    const usecase: TodoUsecase = makeUsecase();
    expect(usecase).toBeDefined();
  });

  it('create returns promise resolving to TodoEntity', async () => {
    const usecase = makeUsecase();
    const app = await createTestApp(usecase);
    
    const todo = await usecase.create({
      appId: app.id,
      title: 'Test Todo',
    });

    expect(todo).toMatchObject({
      id: expect.any(String),
      appId: app.id,
      title: 'Test Todo',
      completed: false,
      createdAt: expect.any(String),
      updatedAt: expect.any(String),
      deletedAt: null,
    });
  });

  it('update with optional completed field', async () => {
    const usecase = makeUsecase();
    const app = await createTestApp(usecase);
    const todo = await usecase.create({
      appId: app.id,
      title: 'Original',
    });

    const updated = await usecase.update({
      appId: app.id,
      todoId: todo.id,
      completed: true,
    });

    expect(updated.completed).toBe(true);
  });
});
```

## 型代入テスト vs 単体テスト

| 側面 | 型代入テスト | 単体 / 統合テスト |
|---|---|---|
| **検証内容** | インターフェース compliance（コンパイル時） | ビジネスロジック（実行時） |
| **実行タイミング** | `tsc --noEmit` 時点で検出 | テスト実行時に検出 |
| **検出可能なエラー** | メソッド名・型ズレ・パラメータ型 | ロジックバグ・戻り値の内容・副作用 |
| **利点** | 素早く検出・CI で 0 秒実行 | 実装の正確性を完全検証 |
| **制限** | 戻り値の「内容」は検証不可 | 型レベルの contract は見えない |

## 型代入テストの活用方法

### 1. pre-commit hook での型チェック

```bash
# package.json
{
  "scripts": {
    "typecheck": "tsc --noEmit"
  },
  "husky": {
    "hooks": {
      "pre-commit": "npm run typecheck"
    }
  }
}
```

コミット前に型チェックを実行 → interface compliance を保証

### 2. vitest での型安全テスト

```typescript
// vitest で型チェック + ロジックテストを同時実施
describe('AppUsecase type-safe integration', () => {
  it('satisfies interface and executes correctly', async () => {
    // 型代入テスト：コンパイルエラーが出ないことを保証
    const usecase: AppUsecase = makeUsecase();

    // ロジックテスト：実行時の正確性を検証
    const app = await usecase.create({ name: 'Test' });
    expect(app.name).toBe('Test');
    expect(app.id).toMatch(UUID_RE);
  });
});
```

## 型代入テストが価値を提供する理由

1. **development-time feedback** — コンパイル時に間違いを検出 → `npm run build` で即座に失敗
2. **refactoring の安全性** — interface を変更するたびに、すべての実装場所がコンパイルエラーで浮き上がる
3. **contract の明確化** — interface 定義が「実装が満たすべき約束」として機能
4. **ドメイン層の型安全性** — business logic のバグを型システムで未然防止

---

## インターフェース契約テストの成果

型代入テストにより：

1. **実装と interface の乖離が検出される** — 早期に・自動的に
2. **デリバリーの品質が上がる** — 「インターフェースをちゃんと実装した」という確信
3. **refactoring が安全** — interface 変更時に影響を受けるすべての実装場所が分かる
4. **team の心理的安全性** — contract が明確 → チームメンバーが interface に信頼を置ける

---

## まとめ

インターフェース契約テストは、単なる「型チェック」ではなく、**ビジネスロジック実装の正確性を型システムで保証する手法**である。

特に Clean Architecture パターンでは、usecase / interactor のインターフェース定義が「層間の契約」として機能する。

型代入テストを習慣づけることで、開発者は「interface に従った実装」を自動的に強制され、バグ検出のコストが大幅に削減される。
