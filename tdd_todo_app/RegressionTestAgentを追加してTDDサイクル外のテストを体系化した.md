# RegressionTestAgentを追加してTDDサイクル外のテストを体系化した

## 対象読者

- マルチエージェント TDD 環境を運用しているエンジニア
- 「実装は終わったのに、安全網となるテストが足りない」と感じたことがある人
- RedAgent / GreenAgent / RefactorAgent の役割分担を理解している人

## この記事が扱う範囲

- **扱う**: RegressionTestAgent の目的・使い分け・テスト選択ルール・完了定義
- **扱わない**: Vitest の使い方、TDD の Red/Green/Refactor サイクルの詳細

---

## 背景：TDD サイクルが埋めない穴

Red/Green/Refactor というTDDの3サイクルは「これから実装する機能」を中心に設計されている。
しかし実際の開発では次のような状況が頻繁に発生する。

- すでに動いている機能にテストがない、または薄い
- バグを修正したが「再発防止テスト」を書いていない
- リファクタリング前の動作を保護するテストが存在しない
- API のレスポンス形状やステータスコードを保証する契約テストがない

これらは TDD の Red フェーズ（実装前に失敗するテストを書く）に属さない。
RedAgent を呼ぶのは概念的に間違いで、かといって誰も担当しないままになってしまう。

---

## 解決策：RegressionTestAgent の追加

`RegressionTestAgent` は **「実装済みコードに対する安全網テストの作成・実行」** に特化したエージェントとして追加された。

```
.github/agents/RegressionTestAgent.agent.md
```

### 役割の定義

> RegressionTestAgent は post-implementation safety-net testing の専門家。  
> TDD の Red フェーズ外で必要なテストを特定・作成・実行し、既存コードへの信頼を高める。

---

## いつ使うか / 使わないか

明確な「使いどき」と「使わないとき」が定義されている点が重要。

### ✅ 使うべき場面

1. 機能は動いているがリグレッション coverage が薄い
2. バグ修正に対して永続的な再発防止テストが必要なとき
3. リファクタリング前に動作保護テストを追加したいとき
4. ルート/ユースケース/リポジトリ境界の統合テストを追加したいとき
5. マージ前にクリティカルフローのスモークテストが必要なとき

### ❌ 使ってはいけない場面

| やりたいこと | 正しいエージェント |
|---|---|
| 新機能の実装前に失敗するテストを書く | `@RedAgent` |
| 失敗しているテストを通す実装コードを書く | `@GreenAgent` |
| 既存テストをそのままに、コードだけ整理する | `@RefactorAgent` |

使い分けが明文化されることで、「どのエージェントに頼めばいいか」という迷いがなくなる。

---

## テスト選択ルール

全てのテストを追加するのではなく、**リスクに見合う最小セット**を優先する。
選択の優先順位は次のとおり。

1. **バグ再現テスト** — 既知の障害や review 指摘に対して
2. **リグレッションテスト** — 変更箇所や壊れやすい動作に対して
3. **統合テスト** — ルート/ユースケース/リポジトリ境界に対して
4. **契約テスト** — レスポンス形状・ステータスコード・外部動作に対して
5. **スモークテスト** — クリティカルな happy path に対して

逆に次のテストは追加しない。

- 既存の assertion と重複し、追加の保護を生まない
- 外部から見える動作ではなく実装の詳細を検証する
- 新しいテストフレームワークの導入が必要になる
- あまりに広すぎて失敗時の診断が困難になる

---

## 実行ルール

バックエンドでのデフォルト検証フローは次のとおり。

```bash
cd backend
npm run lint
npm run typecheck
npm run test
```

「実際にコマンドを実行せずに完了と報告してはならない」というルールが明記されており、
環境の制約でコマンドを実行できない場合は「ブロック理由を明確に報告する」ことが求められる。

---

## 完了の定義

RegressionTestAgent は次の条件が揃って初めて「完了」とみなされる。

- [ ] 必要な非TDDテストが追加されている
- [ ] 追加したテストが実際のリスク/変更スコープと一致している
- [ ] リポジトリのテスト規約に従っている
- [ ] 関連する検証コマンドが実行されている
- [ ] 最終報告に pass/fail/blocked の状態が明記されている
- [ ] ブロックがあれば曖昧にせず具体的に説明されている

---

## 使用例

### 例1：バグ修正後のリグレッションテスト追加

```
@RegressionTestAgent この不具合に対する再発防止テストを作って backend API を検証して
Issue: todo completion flag is lost after update
```

この場合、エージェントは次のようなテストを追加する。

- タイトル更新時に `completed` がリセットされないこと
- App 削除後、その todos が一覧・詳細ルートに露出しないこと
- 重複 App 名の競合が `409` を返すこと

### 例2：リファクタリング後の統合安全テスト追加

```
@RegressionTestAgent backend の最近の変更に必要な回帰テストを追加して実行して
```

ルート/コントローラ/ユースケースの配線が変わった場合、次を検証する。

- コントローラが正しい HTTP ステータスとレスポンス形状を返すこと
- エラーマッピングが `{ success: false, data: null, error }` を生成すること
- ルートエンドポイントとコア CRUD フローがエンドツーエンドで動作すること

## まとめ

| 観点 | Red/Green/Refactor エージェント | RegressionTestAgent |
|---|---|---|
| 対象フェーズ | TDD サイクル内 | TDD サイクル外 |
| テスト作成タイミング | 実装前（Red）または実装後（Refactor） | 実装済みに対して |
| 目的 | 機能の設計・実装 | 既存動作の保護・再発防止 |
| 実行責任 | 各フェーズエージェント | RegressionTestAgent |

TDD サイクルが「新しいコードを正しく作る」ためのプロセスだとすれば、
RegressionTestAgent は「すでにあるコードを壊さないために守る」専門家として機能する。  
役割を分離することで、どのタイミングでどのエージェントを呼ぶべきかが一意に決まり、
テストの抜け漏れを構造的に減らせる。
