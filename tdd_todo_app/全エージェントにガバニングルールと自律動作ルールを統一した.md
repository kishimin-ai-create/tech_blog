# 全エージェントにガバニングルールと自律動作ルールを統一した

## 対象読者

- Copilot エージェントを複数運用しているエンジニア
- 「エージェントがどのルールに従うべきか」を明文化したい人
- エージェントが毎回確認を求めてくる問題を解消したい人

## この記事が扱う範囲

- **扱う**: ガバニングルールテーブルの追加、自律動作ルールの追加、その意図
- **扱わない**: 各ルールファイルの詳細内容、エージェントの実装ロジック

---

## 背景

このリポジトリでは以下のエージェントが定義されている。

```
ArticleWriterAgent / CodeReviewAgent / GreenAgent / OpenApiWriterAgent /
OrchestratorAgent / PullRequestWriterAgent / RedAgent / RefactorAgent /
RegressionTestAgent / ReviewResponseAgent / UIDesignAgent / WorkSummaryAgent
```

以前は「どのルールファイルをエージェントが読むべきか」が明文化されていなかった。
またエージェントが指示を受けるたびに「〜してもよいですか？」と確認を求めることがあり、
自律的な作業の妨げになっていた。

2つのコミットでこの問題を解消した。

- `d15dd83`: 全エージェントにガバニングルールテーブルを追加
- `fd5b4b0` + `fccf128`: 全エージェントに自律動作ルールを追加

---

## ガバニングルールテーブルの追加

各エージェントファイルの末尾に `## 📚 Governing Rules` セクションが追加された。

```markdown
## 📚 Governing Rules

Before acting, read the following rule files and apply them throughout all work:

| Rule File | Applies to |
|---|---|
| `.github/rules/principles.rules.md`       | Core engineering principles |
| `.github/rules/protected-paths.rules.md`  | Files that must not be modified |
| `.github/rules/backend.rules.md`          | Backend architecture — Clean Architecture |
| `.github/rules/frontend.rules.md`         | Frontend architecture — React, Tailwind CSS |
| `.github/rules/typescript.rules.md`       | TypeScript coding standards |
| `.github/rules/test.rules.md`             | Test writing standards |
| `.github/rules/git.rules.md`              | Git workflow rules |
```

エージェントごとに参照するルールファイルが異なる。例えば：

- **UIDesignAgent**: `human-interface-guideline.rules.md` を優先参照
- **RegressionTestAgent**: `test.rules.md` と `test-driven-development.rules.md` を含む
- **ArticleWriterAgent**: `git.rules.md`（エビデンス収集）と `protected-paths.rules.md`

### なぜテーブル形式か

テーブルは「どのルールファイルが何を規定しているか」を一目で把握できる。
エージェントが作業を始める前に参照すべきルールが明確になり、
「ルールを知らなかったために規約を破る」という状況を防ぐ。

## 自律動作ルールの追加

### copilot-instructions.md への追加（`fccf128`）

リポジトリ全体の Copilot 指示ファイル（`.github/instructions/copilot-instructions.md`）に
次のガイドラインが追加された。

```markdown
## 自動化に関するガイドライン

- ユーザーからの指示を受けたら、確認なしに直ちに作業を開始する
- ユーザーの許可を求めることなく、自律的に動作する
```

### 各エージェントへの個別追加（`fd5b4b0`）

9つのエージェント定義ファイルに対し、次の1行が追加された。

```markdown
- Do not ask for permission or confirmation before acting; proceed autonomously.
```

追加対象エージェント：
ArticleWriterAgent / CodeReviewAgent / GreenAgent / OpenApiWriterAgent /
PullRequestWriterAgent / RedAgent / RefactorAgent / WorkSummaryAgent、
および OrchestratorAgent（既存のルール記述を整理する形で更新）

---

## OrchestratorAgent の更新内容

OrchestratorAgent は他のエージェントを調整する役割を持つため、
自律動作ルールの追加に加えて既存の「コードを自分では書かない」ルールが整理された。

変更前後のルール例（`fd5b4b0`）：

```diff
- ## Rules
- - Do not write code yourself.
- - Delegate to RedAgent, GreenAgent, RefactorAgent.
  
+ ## Strict Rules
+ 1. Never write implementation code yourself — always delegate.
+ 2. Do not ask for permission before orchestrating the next step.
+ 3. If a step fails, report clearly and wait for human guidance.
```

---

## 2つのルールが組み合わさることで生まれる効果

| 変更前 | 変更後 |
|---|---|
| エージェントがどのルールを読むか曖昧 | Governing Rules テーブルで明示 |
| エージェントが毎回確認を求める | 自律動作ルールにより即座に作業開始 |
| ルール違反が「知らなかった」で起きる | 作業前にルールを読むことが義務化 |
| 新しいエージェントを追加するたびに手順が属人化 | テーブル形式で統一・再現可能 |

---

## まとめ

「全エージェントに同じ形式でガバニングルールを追加する」という機械的に見える変更だが、
その効果は大きい。

- **どのルールに従うか**が明文化され、エージェントの動作が予測可能になる
- **確認を求めない**というルールにより、エージェントが指示に対して即座に行動する
- 新しいエージェントを追加する際も、同じテーブル形式に従えばよいという規範が確立された

規則の数を増やすことより、「どの規則をどのエージェントが読むか」を整理することの方が重要だった。
