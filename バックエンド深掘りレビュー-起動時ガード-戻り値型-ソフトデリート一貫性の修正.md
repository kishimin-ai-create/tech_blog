## 対象読者

- TypeScript + Hono + Clean Architecture 構成のバックエンドを保守しているエンジニア
- コードレビューで「なぜこの修正が必要なのか」まで理解したい方
- ソフトデリート実装の落とし穴や、起動時の環境変数チェック設計に興味がある方

## 修正 1: `mysql-client.ts` — 認証情報欠如時の起動ガード

### 問題

```ts
// 修正前
user:     process.env.DB_USERNAME ?? 'root',
password: process.env.DB_PASSWORD ?? '',
```

`DB_USERNAME` や `DB_PASSWORD` が環境変数に設定されていない場合、コードは `root` / 空パスワードで接続プールを作成していた。設定ミスのコンテナや Secret の注入忘れが起きたとき、MySQL インスタンスの設定次第では **スーパーユーザー権限でサイレント接続が成功**してしまう。問題が起きているかどうかログからまったく分からない。

### 修正

```ts
// backend/src/infrastructure/mysql-client.ts（修正後）
/**
 * Creates a MySQL connection pool from environment variables.
 * Throws a startup error if required credentials are absent.
 */
export function createMysqlPool(): MysqlPool {
  const user = process.env.DB_USERNAME;
  const password = process.env.DB_PASSWORD;
  if (!user || password === undefined) {
    throw new Error(
      'DB_USERNAME and DB_PASSWORD environment variables are required',
    );
  }
  return mysql.createPool({
    host: process.env.DB_HOST ?? '127.0.0.1',
    port: Number(process.env.DB_PORT ?? '3306'),
    database: process.env.DB_DATABASE ?? 'TDDTodoAppDB',
    user,
    password,
    // ...
  });
}
```

**設計のポイント**:
- `password` のチェックは `!password` ではなく `password === undefined` を使う。空文字列 `''` は「意図的に設定された空パスワード」と区別できないため、`undefined`（未設定）のみを拒否する
- フォールバックを排除することで、環境変数の欠如が **プロセス起動時に即座に爆発**する。問題の発見が接続試行時から起動時に早まり、デバッグが簡単になる
- `DB_HOST`、`DB_PORT`、`DB_DATABASE` はインフラ構成値であり、環境差があっても安全なデフォルトが存在するため `??` フォールバックを維持した

全 387 件のテストがパスしている。

## 修正 2: 3 つのファクトリー関数に明示的な戻り値型を追加

### 問題

TypeScript ルールでは、公開（エクスポート）関数にはすべて明示的な戻り値型が必要とされている。しかし以下の 3 関数が戻り値型を省略していた:

| ファイル | 関数 |
|----------|------|
| `infrastructure/hono-app.ts` | `createHonoApp(dependencies)` |
| `infrastructure/registry.ts` | `createBackendRegistry()` |
| `infrastructure/mysql-registry.ts` | `createMysqlBackendRegistry()` |

明示的な戻り値型がないと、リファクタリングで戻り値の構造が変わっても（フィールドの追加・削除、`Hono` 型の変更など）**呼び出しサイトの型エラーとして検出されない**。問題は遠く離れたファイルで初めてビルドエラーとして顕在化し、原因の特定に時間がかかる。

### 修正

```ts
// 例: registry.ts（修正後）
type BackendRegistry = { app: Hono; clearStorage: () => void };

export function createBackendRegistry(): BackendRegistry { ... }
```

```ts
// 例: mysql-registry.ts（修正後）
type MysqlBackendRegistry = { app: Hono };

export function createMysqlBackendRegistry(): MysqlBackendRegistry { ... }
```

```ts
// 例: hono-app.ts（修正後）
export function createHonoApp(dependencies: HonoAppDependencies): Hono { ... }
```

**設計のポイント**:
- `BackendRegistry` と `MysqlBackendRegistry` のように戻り値型に名前を付けることで、型が「自己文書化」される
- `clearStorage` が `BackendRegistry` にしか存在しないことが型レベルで明確になり、テスト用の関数がプロダクション向けレジストリには持ち込まれていないことが保証される
- 型チェックと全 387 件のテストがパスしている

## 修正 3: ソフトデリート時に `updatedAt` を同時更新する

### 問題

アプリや Todo をソフトデリートするとき、`deletedAt` はセットされるが `updatedAt` が更新されていなかった。

```ts
// app-interactor.ts 修正前
const deletedApp: AppEntity = { ...app, deletedAt };   // updatedAt はそのまま

// todo-interactor.ts 修正前
const deletedTodo: TodoEntity = { ...todo, deletedAt: now() };  // updatedAt はそのまま
```

このとき `updatedAt < deletedAt` という状態が生まれる。`updatedAt` を変更カーソルとして使うクライアント（同期ポーリングなど）は「最終更新以降に何も変わっていない」と判断し、**ソフトデリートの変更を見逃す**可能性がある。

### 修正

すべてのソフトデリートパスで `updatedAt` を `deletedAt` と同じタイムスタンプにセットする:

```ts
// app-interactor.ts — 直接削除（修正後）
const deletedAt = now();
const deletedApp: AppEntity = { ...app, updatedAt: deletedAt, deletedAt };

// app-interactor.ts — カスケード削除（修正後）
for (const todo of todos) {
  await todoRepository.save({ ...todo, updatedAt: deletedAt, deletedAt });
}

// todo-interactor.ts — 直接削除（修正後）
const deletedAt = now();
const deletedTodo: TodoEntity = { ...todo, updatedAt: deletedAt, deletedAt };
```

**設計のポイント**:
- 「ソフトデリートも更新の一種である」という意味論的な一貫性を保つ
- `deletedAt` と `updatedAt` を同じ `now()` 呼び出し結果で揃えることで、タイムスタンプの順序関係が常に `updatedAt === deletedAt` になる
- カスケード削除では親のタイムスタンプ変数 `deletedAt` を子の `updatedAt` にも流すことで、バッチ全体が同一タイムスタンプを持つ

全 387 件のテストがパスしている。

## Reply-only: 今回対応しなかった指摘と今後の方針

### カスケードデリートの非アトミック問題

`app-interactor.ts` の `remove` 関数は次の 3 ステップで動く:

```ts
// step 1: App に deletedAt をセット
await appRepository.save(deletedApp);

// step 2: 紐づく Todo 一覧を取得
const todos = await todoRepository.listActiveByAppId(app.id);

// step 3: Todo を 1 件ずつ更新
for (const todo of todos) {
  await todoRepository.save({ ...todo, updatedAt: deletedAt, deletedAt });
}
```

step 1 と step 3 の間で MySQL エラーやプロセスクラッシュが起きると、App の `deletedAt` はセットされているが一部の Todo はアクティブのまま残る。これらの Todo は「削除されたアプリに紐づく孤立レコード」になり、復旧パスがない。

**対応しなかった理由**: トランザクション境界を設けるには、リポジトリインターフェースに `withTransaction(callback)` ポートを追加するか、MySQL レイヤーでバルク UPDATE を発行する設計変更が必要になる。どちらも Clean Architecture の境界に触る大きな変更であり、アプリ・Todo 双方のリポジトリ実装（インメモリ・MySQL）すべてへの影響を伴う。現状、全テストはインメモリ実装で動いており Map 操作は途中失敗しないため影響範囲はゼロ。MySQL パスはエラーが `REPOSITORY_ERROR` として顕在化するため運用上は検知可能。**トランザクションポートの導入を後続タスクとして追跡する**。

### `clearStorage` と `NODE_ENV` 分岐が `index.ts` に混在している問題

`backend/src/index.ts` はプロダクションサーバーとテストの両方がインポートするが、`NODE_ENV === 'test'` の分岐と `clearStorage` エクスポートが同居している。これはテスト専用のインフラコードがプロダクションバンドルに含まれることを意味する。

**対応しなかった理由**: `index.ts` を解体して `server.ts` をコンポジションルートにする変更は、`clearStorage` の参照先変更・テストヘルパー層の再配線・既存のインポートサイト確認など非自明なリスクを伴う。`NODE_ENV === 'test'` ガードが MySQL 接続の誤起動を防いでいるため即座のプロダクション影響は限定的。**コンポジションルート整理として別 PR で追跡する**。

### 二重バリデーション（ランタイム vs Zod）の乖離リスク

`controllers/request-validation.ts`（実際に実行されるバリデーション）と `controllers/schemas.ts`（OpenAPI ドキュメント用 Zod スキーマ）が別々に実装されており、どちらかを更新しても他方は自動的に追随しない。

**対応しなかった理由**: `request-validation.ts` を Zod の `safeParse` に置き換えるには、Zod のエラー形状を既存の `AppError` コードにマッピングし、エラープレゼンター層を更新し、30 件以上のバリデーションテストケースを確認する必要がある。デュアルバリデーション統合は専用 PR として追跡済み。

### テストファイルの `as` 型アサーション

`as` キャストに説明コメントを付けるルールが TypeScript ルールに定められているが、テストファイル全体で数十カ所の無注釈 `as` キャストが存在する。レスポンス型解析ヘルパー（Zod `safeParse` を使う）を導入するか、全箇所にコメントを追記する対応が必要だが、デュアルバリデーション統合と合わせて実施するほうが効率的なため後続に委ねる。

## まとめ

| 修正 | ファイル | 優先度 |
|------|----------|--------|
| 認証情報欠如時の起動ガード | `infrastructure/mysql-client.ts` | P2 |
| ファクトリー関数の明示的戻り値型 | `hono-app.ts` / `registry.ts` / `mysql-registry.ts` | P2 |
| ソフトデリート時の `updatedAt` 同時更新 | `services/app-interactor.ts` / `services/todo-interactor.ts` | P3 |

今回の修正から得られる教訓は 3 つ:

1. **環境変数の欠如は「起動時に爆発」させるべき**: サイレントフォールバックは問題の発見を遅らせ、最悪の場合セキュリティリスクになる
2. **公開関数の戻り値型は省略しない**: 型推論に頼ると、リファクタリング時の型エラーが呼び出しサイトに伝播せず問題を見落とす
3. **タイムスタンプは意味論的に一貫させる**: ソフトデリートも「更新」であり、`updatedAt` を変更カーソルとして使う設計においては `deletedAt` と揃える必要がある

reply-only としたトランザクション対応とコンポジションルート整理は、それぞれ独立した PR として今後実施する予定だ。
